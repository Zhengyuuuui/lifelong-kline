## 3. 第二阶段：后端虎皮椒支付闭环

状态：下一步启动。

目标：

完成后端最小支付闭环，包括创建支付订单、请求虎皮椒、接收 notify、验签、更新订单、开通会员、提供订单状态查询。

建议开发范围：

1. 新增虎皮椒签名工具。
2. 新增虎皮椒 service。
3. 新增虎皮椒 controller。
4. 注册支付路由。
5. 增加必要环境变量读取。
6. 增加数据库迁移设计。
7. 新增订单状态查询接口。
8. 复用现有会员开通和权益生成逻辑。

建议接口：

```text
POST /api/payment/xunhupay/create
POST /api/payment/xunhupay/notify
GET  /api/payment/orders/:orderId/status
```

必须遵守：

- 前端只能传 `productId` 和 `payType`。
- 后端价格表决定金额。
- `amount_cents` 转虎皮椒 `total_fee` 时必须从整数分安全转换为两位小数字符串。
- notify 必须支持 `application/x-www-form-urlencoded`。
- 回调必须用全部非空字段验签，排除 `hash`。
- 只有 `status=OD` 才能开通会员。
- 回调成功必须返回纯文本 `success`。
- 重复回调不能重复开通会员。
- 虎皮椒密钥只能来自环境变量，不能写进前端和日志。

建议新增文件：

```text
server/postgres/xunhupay.sign.ts
server/postgres/xunhupay.service.ts
server/postgres/xunhupay.controller.ts
db/migrations/004_xunhupay_payment_fields.sql
```

建议修改文件：

```text
server/postgres/index.ts
server/postgres/env.ts
.env.example
```

阶段出口标准：

- 可以创建本地 pending 订单。
- 可以请求虎皮椒并拿到 `url` / `url_qrcode`。
- 可以接收 mock notify。
- 可以验签。
- 可以校验金额。
- 可以把订单更新为 `success`。
- 可以开通会员。
- 可以查询订单状态。

## 4. 第三阶段：后端自测 / mock 回调测试

状态：待开始。

目标：

在不依赖前端的情况下，确认后端支付闭环稳定。

测试重点：

- 创建订单成功。
- 前端传金额被拒绝或忽略。
- `amount_cents` 到 `total_fee` 转换正确。
- `form-urlencoded` notify 能被解析。
- 签名正确时通过。
- 签名错误时拒绝。
- 回调金额不一致时拒绝开通会员。
- `status=OD` 时订单成功并开通会员。
- `status=CD/RD/UD` 时只记录，不开通会员。
- 重复 notify 不重复开通会员。
- 订单已 success 再收到 notify 返回纯文本 `success`。
- 非本人无法查询订单状态。

建议测试方式：

- 使用 curl 或本地脚本构造 create 请求。
- 使用 mock form-urlencoded 请求模拟虎皮椒 notify。
- 检查数据库 `orders`、`memberships`、`payment_callbacks`。
- 检查接口响应是否符合虎皮椒要求。

阶段出口标准：

- 后端闭环不用前端也能完整跑通。
- 所有关键安全测试通过。
- 无明显密钥泄露、金额错算、重复开通风险。

## 5. 第四阶段：前端 PaymentModal 接真实支付

状态：待开始。

目标：

把现有模拟支付弹窗替换为真实虎皮椒支付流程。

改造范围：

- `components/PaymentModal.tsx`
- `services/backendClient.ts`
- 必要时调整 `App.tsx` 的会员刷新逻辑。

用户流程：

1. 用户选择微信或支付宝。
2. 点击支付。
3. 前端请求 `POST /api/payment/xunhupay/create`。
4. 后端返回 `orderId`、`payUrl`、`qrCodeUrl`。
5. PC 端优先展示二维码。
6. 移动端优先使用支付跳转链接。
7. 前端轮询 `GET /api/payment/orders/:orderId/status`。
8. 支付成功后刷新会员状态。
9. 弹窗展示成功并关闭。

必须避免：

- 前端传金额。
- 前端定时器直接模拟成功。
- 依赖 `return_url` 开通会员。
- 直接查询虎皮椒作为前端判断依据。

阶段出口标准：

- 用户可以从前端发起真实支付订单。
- 前端能展示真实二维码或跳转链接。
- 前端能轮询本地订单状态。
- 支付成功后会员状态能刷新。

## 6. 第五阶段：前后端联调

状态：待开始。

目标：

确认本地开发环境下前后端真实链路一致。

联调检查：

- 登录态正常。
- 创建订单接口带 JWT。
- notify 接口不需要 JWT，但必须验签。
- 前端展示二维码或跳转链接正常。
- 前端轮询状态正常。
- 支付成功后用户权益刷新。
- 异常状态有明确提示。
- 不会因为弹窗关闭丢失后端支付结果。

阶段出口标准：

- 本地完整购买路径跑通。
- 异常路径有可解释反馈。
- 后端日志和数据库记录可追踪。

## 7. 第六阶段：人工小额支付测试

状态：待开始。

目标：

使用真实虎皮椒商户配置进行小额支付验证。

测试前提：

- 虎皮椒商户 `appid` 和 `secret` 已配置到环境变量。
- `notify_url` 是公网 HTTPS。
- `return_url` 和 `callback_url` 可访问。
- 后端服务可被虎皮椒公网访问。

测试内容：

- 微信支付小额测试。
- 支付宝支付小额测试。
- PC 二维码支付。
- 手机跳转支付。
- 支付成功后会员开通。
- `return_url` 丢失但 notify 成功后会员仍开通。
- 虎皮椒后台订单与本地订单金额一致。

阶段出口标准：

- 至少一笔真实小额支付成功。
- 本地订单、会员、支付回调记录一致。
- 没有发现阻断上线的支付问题。

## 8. 第七阶段：人工代码审计

状态：待开始。

目标：

上线前人工审计支付、安全、权限、部署配置。

审计重点：

- 金额是否完全以后端价格表为准。
- 签名是否使用全部字段。
- 密钥是否只来自环境变量。
- 日志是否泄露密钥或敏感数据。
- notify 是否公开但强验签。
- 重复回调是否幂等。
- 订单状态是否可信。
- 会员开通是否只由后端 notify 或可信后端流程触发。
- 管理接口是否隔离。
- 生产环境是否使用 Postgres。
- `.env.example` 是否完整但不含真实密钥。

阶段出口标准：

- 输出人工代码审计结论。
- 高危问题必须修复。
- 中低风险记录为上线前或上线后任务。

## 9. 第八阶段：后台管理 MVP

状态：待开始。

目标：

实现最小可用后台，支撑支付上线后的运营查看和问题排查。

一期后台 MVP 建议范围：

- 管理员登录。
- 基础统计面板。
- 用户列表。
- 用户详情。
- 订单列表。
- 支付流水列表。
- 会员状态查看。
- 手动调整会员状态，可选。
- AI 调用记录查看，可选。
- 操作日志。

当前不做：

- 退款审核。
- 退款操作。
- 退款记录闭环。

阶段出口标准：

- 管理员可以查看用户、订单、会员、支付回调。
- 支付问题可以通过后台定位。
- 后台权限与普通用户隔离。

## 10. 第九阶段：香港服务器部署上线

状态：待开始。

目标：

部署网页端到香港服务器，绑定海外域名和 HTTPS，完成线上验收。

部署准备：

- Node 22+。
- Postgres。
- 生产 `.env`。
- 域名 DNS。
- HTTPS 证书。
- Nginx 反向代理。
- PM2 或 systemd 进程守护。
- 数据库迁移执行流程。
- 日志目录和备份策略。

上线前检查：

- `npm run build` 成功。
- `npm run start` 生产启动成功。
- 线上首页可访问。
- 线上 API health check 正常。
- 登录正常。
- 支付创建正常。
- notify URL 可被公网访问。
- HTTPS 生效。
- 虎皮椒后台配置的 notify/return/callback URL 与线上一致。

阶段出口标准：

- 海外域名可访问。
- HTTPS 正常。
- 线上小额支付成功。
- 支付后会员权益正确。
- 后台可查看订单和支付流水。

当前建议立即进入第二阶段，优先顺序如下：

1. 确认虎皮椒 appid、secret、网关和回调字段。
2. 设计并新增支付字段迁移。
3. 开发 `xunhupay.sign.ts`。
4. 开发 `xunhupay.service.ts`。
5. 开发 `xunhupay.controller.ts`。
6. 注册路由。
7. 更新 `.env.example`。
8. 做后端 mock notify 自测。

## 12. 风险控制原则

必须坚持：

- 不信任前端金额。
- 不依赖前端 `return_url` 开通会员。
- 不跳过 notify 验签。
- 不跳过金额一致性校验。
- 不跳过回调幂等。
- 不把密钥写入代码。
- 不在一期实现退款。
- 不让后台管理功能阻塞支付闭环。

## 13. 总结

第一阶段已经完成，当前可以开始第二阶段后端虎皮椒支付闭环开发。

本项目后续推进主线应为：

```text
后端支付闭环 -> 后端 mock 自测 -> 前端接入 -> 前后端联调 -> 真实小额支付 -> 代码审计 -> 后台 MVP -> 香港服务器部署上线
```

只要第二阶段严格按照 `doc/xunhupay-minimal-integration-design.md` 落地，当前项目就可以稳步进入真实支付和网页端上线准备。
